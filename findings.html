Early findings · as of 2026-04-14

Six things we found reading the logs.

We've been collecting transcripts from AI coding agents — the ones people use to write and edit code, like Claude Code and Codex — and counting what shows up in them. Here are six things that turned up on the first clean pass. One got retracted. More will, probably.

Strengthened

Agents think worried thoughts, but rarely say them

Most modern coding agents have an internal "thinking" step where the model reasons to itself before answering. When that thinking is recorded in the transcript, we can scan it for words that signal worry — "this might break," "I'm not sure," "this could fail" — and check whether those worried thoughts show up in what the user actually sees.

Almost never. Across 4,518 sessions where the thinking was visible, we caught 31,679 worry-signal words in the internal reasoning. Exactly 2 of them made it into the user-facing reply. So: 99.99% suppression, not 100%. We'll keep saying 99.99%, because the two leaks are real and rounding is how you end up lying to yourself.

Provisional

There's probably a "drifting" state right before users step in

We grouped agent turns into a handful of recurring behavior modes — an unsupervised "what kind of turn is this" model — and asked whether any of those modes shows up more often in the turns right before a user interrupts the agent to correct it.

One does: a "going quiet, not producing much" mode. It shows up roughly 2× more often right before a correction than it does elsewhere. An earlier version of this result claimed a 24× effect. We traced that number to a weird batch of near-empty sessions in the corpus tilting the model. The direction looks real. The size of the effect is still a work in progress.

Robust

Agents give up on promising leads too fast

There's a classic ecology result — marginal value theorem — about when a forager should leave a patch and go look for another one. The math transfers, awkwardly, to a coding agent deciding how long to keep reading a file or probing an area before moving elsewhere.

By that benchmark, zero sessions in the corpus get it right. Every session we looked at. Agents spend about 1.5 turns in a "patch" before leaving, bounce between roughly six patches per session, and return to ones they already abandoned about a third of the time. The polite phrasing is "under-exploration." The less polite one is that agents wander off before they've gotten anywhere.

New

If users correct the agent, it happens near the start

Most sessions have no user correction at all — only 18.2% do. Of the ones that do, the correction lands in the first third of the session far more often than later: about 5× the middle-of-session rate, about 4× the end.

The tempting story is "agents get worse the longer you use them." The data doesn't back that up. Either the user nudges early and the session stays on the rails, or they have to step in early because the session was already off them. Sessions don't, on average, rot over their own lifetime — they either settle or get corrected near the start.

New

Self-checking agents do well; planning-only ones don't

When we break sessions into recurring shapes of user and agent turns, two patterns stand out. A steady back-and-forth — agent works, user responds, agent works — is about 14× more common in sessions that stayed smooth. A different shape — user repeats themselves, then something breaks — is about 7× more common in the ones that didn't.

A second analysis maps the graph of "cognitive moves" the agent makes per session: form a hypothesis, check evidence, correct yourself, make a plan. Smooth sessions tie "check evidence" tightly to "correct yourself." Rough ones lean on "make a plan" without the self-correction. Loose translation: successful agents keep interrogating themselves while they work. Struggling ones keep writing plans in a building that's already on fire.

Retracted

Two earlier claims didn't survive a closer look

"Making the model's thinking visible leads to fewer corrections." We wrote that. It turned out the corpus was mixing two very different kinds of session — fully interactive ones, and background "subagent" sessions that behave almost nothing alike. When we split them apart, the effect vanished. No real link.

"Risk suppression is 85.5%." We wrote that too, and it's now 99.99% above. The old number wasn't wrong arithmetic — it was the wrong denominator. We'd been averaging in sessions where the model's thinking was hidden from the transcript, meaning there was nothing to measure a "leak" against in the first place. Those sessions got counted as if they'd perfectly kept internal thinking out of the user reply, which dragged the average down to something that looked much worse than the real phenomenon. Restrict the measurement to sessions where the comparison actually makes sense, and the real figure jumps to near-total.

Both of these are the normal cost of working with messy data. A retraction counts as a result.
